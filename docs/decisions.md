# Decisions (ADR-lite)

One line per judgment call. Newest at the bottom.

- 2026-07-19 — Ingestion keeps ALL 197 raw columns as-is (raw zone is raw); column curation to the SMART subset happens in dbt staging, not here (Blueprint Section 4 step 3).
- 2026-07-19 — Parquet is Hive-partitioned by `quarter` (`data/parquet/quarter=YYYYQN/`) to enable partition pruning in Phase 3; the `quarter` column is encoded in the path, not stored in the files.
- 2026-07-19 — Only `date`, `failure`, `capacity_bytes` are explicitly cast at ingestion (DATE / TINYINT / BIGINT). All other columns keep DuckDB's inferred types from a full-file type scan (`sample_size=-1`) so sparse SMART columns are typed correctly across schema drift.
- 2026-07-19 — Ingestion parquet compression is zstd (better ratio than snappy, fast enough) to stay within the ~15GB disk budget.
- 2026-07-19 — Q1 2026 actual row count (30,597,484) exceeded blueprint's ~27M estimate. No issue found; pre-run estimate was rough. Real number is what's used going forward in all benchmarks/reconciliation/resume citations.
- 2026-07-19 — dbt project created by writing config files directly (not interactive `dbt init`), because the `dbt/` folder skeleton from the Phase 1 scaffold already existed; `dbt init` would have nested a second project under it. `dbt debug` = all checks passed confirms the hand-authored setup is valid.
- 2026-07-19 — Parquet declared as a dbt-duckdb external source via `meta.external_location: read_parquet('../data/parquet/quarter=*/*.parquet', hive_partitioning = true)`. Relative path resolves against the `dbt/` working dir, so dbt must be invoked from `dbt/`. hive_partitioning exposes the path-encoded `quarter` column to the models.
- 2026-07-19 — Staging materialized as a `view` (not table): it is a thin rename/cast contract over the external Parquet, so a table copy would just duplicate 30.6M rows on disk for no query benefit at this scale. Revisit if downstream repeatedly rescans it.
- 2026-07-19 — `capacity_bytes` sentinel (<= 0) is FLAGGED at staging via a boolean `is_capacity_sentinel`, not repaired. Repair to modal capacity is deferred to dim_drive (Blueprint 5.2). Keeps staging a pure contract layer with no business logic.
- 2026-07-19 — Staging carries no grain-uniqueness test; per Blueprint 5.7 that test lives on fct_drive_daily. Any (serial_number, snapshot_date) dedupe is deferred to the fact layer so staging stays a faithful 1:1 mirror of the raw Parquet (row-parity test enforces this).
- 2026-07-19 — Source freshness declared with generous thresholds (warn 120d / error 400d) on `date`. The dataset is quarterly and historical, so freshness documents cadence rather than gating; it is not part of the `dbt build` completion gate.
- 2026-07-19 — int_drive_spans materialized as `table` (Phase 2). It collapses 30.6M drive-days to one row per drive (351,095) and is reused by dim_drive + cohort/survival marts, so a view would recompute the aggregation on every reference. Set via dbt_project.yml `intermediate: +materialized: table`.
- 2026-07-19 — Censoring `active` is anchored to the dataset-wide `max(snapshot_date)` (currently 2026-03-31), not a hardcoded quarter-end, so the classification generalizes automatically as Phase 3 adds quarters.
- 2026-07-19 — Censoring CASE evaluates `failed` before `active`: a drive that fails on the dataset's max date is `failed`, not `active`. Failure outcome takes precedence over the still-observed anchor.
- 2026-07-19 — `observed_days` uses count(*) of daily rows. Verified 0 duplicate (serial_number, snapshot_date) rows in staging, so count(*) == distinct days; no dedupe needed at this layer (grain uniqueness enforced downstream on the fact).
- 2026-07-19 — int_drive_spans censoring sanity (Q1 2026): 351,095 drives = 345,638 active + 4,427 exited_without_failure + 1,030 failed. `failed` count reconciles exactly to the 1,030 raw failure events from Phase 1; observed_days range 1–90 (Q1 has 90 days).
- 2026-07-19 — Some drives show observed_days less than calendar span (first_seen to last_seen) despite continuous presence at both endpoints (e.g. ZA171SP3: 83 observed days over a 90-day span). Reflects real gaps in Backblaze's daily source data, not a staging/ingestion defect (grain check confirms no duplicate/dropped rows). observed_days intentionally counts actual daily observations, which is the correct denominator for AFR-style drive-day calculations.
- 2026-07-19 — Dimensions (dim_drive, dim_model, dim_date) materialized as `table` (dbt_project.yml `dims: +materialized: table`): small, read repeatedly by facts/marts.
- 2026-07-19 — PERF/CORRECTNESS: capacity repair and per-drive model use `max()` (over non-sentinel rows), NOT `mode()`. Three concurrent holistic mode() aggregations over the 30.6M-row staging view caused `dbt build` to hang (>8 min, killed). Verified model and non-sentinel capacity are constant per drive and per model (0 drives/models with >1 distinct value), so max() returns that single value == the modal value while streaming. Build dropped to ~22s.
- 2026-07-19 — Manufacturer is parsed from the model-string prefix via a seed (seeds/seed_manufacturer.csv) using a LONGEST-prefix case-sensitive LIKE match in dim_model, so 'WDC' wins over 'WD'. Unmatched models resolve to 'Unknown' (a signal to extend the seed). Q1 2026: all 80 models matched, 0 Unknown.
- 2026-07-19 — Hitachi is folded into manufacturer 'HGST' in the seed (HGST = the former Hitachi Global Storage Technologies, later acquired by Western Digital). Kept distinct from 'Western Digital' to match Backblaze's own reporting convention. Revisit if a report groups them.
- 2026-07-19 — dim_date generated with a native DuckDB date spine (generate_series over min..max snapshot_date), not dbt_utils.date_spine, to avoid adding a package dependency for one small model. quarter_label ('YYYYQn') deliberately matches the Parquet partition value so dim_date joins cleanly to the raw partition and quarterly AFR grouping.
- 2026-07-19 — Dimension sanity (Q1 2026): dim_drive 351,095 rows; dim_date 90 days (2026-01-01..2026-03-31, 1 quarter); dim_model 80 models. Manufacturer fleet counts sum exactly to 351,095 (each drive maps to one model): Toshiba 117,902 / Seagate 116,850 / Western Digital 88,089 / HGST 27,005 / Crucial 718 / Dell 431 / Micron 92 / Samsung 6 / Intel 2.
- 2026-07-20 — fct_drive_daily materialized as `table` in Tier 1 (dbt_project.yml `facts: +materialized: table`); Phase 3 converts it to incremental. Its surrogate `drive_day_key = serial_number||'|'||snapshot_date` carries the grain-uniqueness test (Gate 3) — deliberately deferred from staging to land on the fact. Verified count(*) == count(distinct key) == 30,597,484.
- 2026-07-20 — fct_failures built from `stg_drive_stats where failure_flag = 1` (NOT from int_drive_spans where censoring_status='failed'). Rationale: failure_flag=1 rows ARE the event grain and preserve the SMART sensor readings AS OF the failure day (useful for failure-signature analysis), whereas int_drive_spans is a drive-level summary without day-level SMART. Both sources agree exactly (1,030), so this is a richer representation of the same events, not a different count.
- 2026-07-20 — Surrogate keys use a plain string concat (serial|date), not md5 or dbt_utils.generate_surrogate_key: avoids a package dependency and hashing cost over 30.6M rows while remaining unique (verified). Revisit if a fixed-width key is needed on Databricks.
- 2026-07-20 — Fact reconciliation sanity (Q1 2026): fct_drive_daily = 30,597,484 rows == raw Parquet (Gate 2 holds through the fact); failures triple-reconcile at 1,030 (fct_failures == int_drive_spans 'failed' == sum of fct_drive_daily.failure_flag); 0 fct_failures dates orphaned from dim_date.
- 2026-07-20 — AFR denominator uses count of fct_drive_daily rows per model PER QUARTER (joined through dim_date), NOT sum(dim_drive.drive_days). dim_drive.drive_days is a drive's lifetime across all loaded quarters; per-quarter counting is the correct denominator and generalizes when Phase 3 adds quarters. Numerically identical with one quarter loaded.
- 2026-07-20 — mart_model_afr_quarterly threshold = drive_count > 100 AND drive_days > 10,000 (Backblaze's published Q1 2026 cutoff, confirmed by owner — not invented). Models below it are excluded from AFR reporting as statistically unstable.
- 2026-07-20 — AFR reasonableness bound (0-50%) is a singular test at severity=warn, not error: a real spike on a small population (e.g. ST16000NM000J 3.61%) is legitimate, not a data defect. Build had 0 warnings (max AFR observed 4.9%).
- 2026-07-20 — AFR RECONCILIATION vs Backblaze published Q1 2026: 33/33 models match EXACTLY on drive_days, failures, and AFR%. Fleet drive-days (30,203,180), failures (1,030), and AFR (1.24%) match exactly; sum of per-model drive_count (346,596) matches the published table's column sum exactly. This is the project's ground-truth verification (Gate 4 PASS). Details in reconciliation_log.md.
- 2026-07-20 — EXPLAINED MISMATCH (drive count): raw distinct drives 351,095 vs Backblaze headline analyzed 341,263. Cause is scope/definition, not error: (a) my mart includes 9 SSD/boot models (3,506 drives, 0 failures) that Backblaze excludes from its HDD table (~3,907 boot + 492 non-qualifying per their report); (b) Backblaze's headline count differs from the sum of its own per-model table (346,596), which my mart matches exactly; (c) remaining drives are in HDD models below threshold, excluded by both. Substantive metrics (drive-days/failures/AFR) reconcile exactly, so failure accounting is verified.
- 2026-07-20 — MODEL-NAMING difference (confirmed, documented, not fixed this checkpoint): Backblaze's source data labels one physical model inconsistently as both "WUH721816ALE6L4" (476 drives) and "WDC WUH721816ALE6L4" (26,325 drives); Backblaze merges them in reporting (26,801). My staging keeps raw model strings, so the mart has two rows for this model. Also, my model labels retain manufacturer prefixes (HGST/TOSHIBA/WDC ) whereas Backblaze's table strips them — cosmetic. Numbers reconcile exactly once normalized. Candidate for a future dim_model model-string normalization (would also merge the WUH split). Left as-is now: it is an honest, explainable data-quality nuance rather than a silent transform.
- 2026-07-20 — mart does not filter drive type, so SSD/boot models pass the threshold and appear in the mart (see reconciliation_log.md). Backblaze's HDD report excludes them. Left in for now (they carry 0 failures and negligibly dilute fleet AFR: whole-mart 1.23% vs HDD-only 1.24%); a drive-type/is_boot flag is a candidate dim_drive enhancement if a strict HDD-only mart is wanted.
- 2026-07-20 — Backblaze's Q1 2026 report noted an unnamed model with two overlapping mechanical issues (mitigated mid-quarter). Checked: since all 33 models match the published AFR exactly, no per-model AFR deviates from Backblaze's numbers — our figures already equal their post-mitigation published values, so no adjustment was needed or made.
- 2026-07-20 — DRIVE-TYPE EXCLUSION uses a genuine source attribute, not a hardcoded model list. Raw data has NO media-type column, but `pod_slot_num` cleanly separates data vs boot drives: the 9 SSD/boot models are 100% NULL slot; the 34 data-model rows are ~99.9% populated (0-59). This is Backblaze's actual criterion (boot-drive exclusion) — and it correctly excludes the two ROTATIONAL boot HDDs (ST500LM030, MQ01ABF050) that a naive SSD-vs-HDD media filter would wrongly keep. Added `pod_slot_num` to the staging contract; dim_model derives `drive_type` = 'data' if majority of a model's rows carry a slot, else 'boot'. mart scopes to drive_type='data' via the dim attribute (not a model-name filter in the mart).
- 2026-07-20 — MODEL-NAMING MERGE fixed upstream via a `normalize_model` macro that strips WD's redundant 'WDC ' corporate prefix (Backblaze labels the same physical model both 'WDC WUH721816ALE6L4' and bare 'WUH721816ALE6L4'; its own tables omit the prefix). Applied in dim_drive and dim_model (NOT the mart) so the fix lives once and every downstream model (incl. Phase 4 marts) inherits it. Manufacturer parsing still resolves (WD family codes WUH/WDS remain in the seed); verified 0 Unknown. dim_model dropped 80→79 models (the split merged to one).
- 2026-07-20 — With both fixes upstream, mart_model_afr_quarterly needs no special-case filtering of its own: querying the corrected dims/facts it returns 33 models / 30,203,180 drive-days / 1,030 failures / 1.24% fleet AFR directly, and all 33 match Backblaze's published Q1 2026 table EXACTLY on drive_count/drive_days/failures/AFR (33/33, verified digit-by-digit). This supersedes the earlier "superset + post-hoc scoping" state. The 341,263 headline vs 346,596 (sum of per-model drive_count) difference remains a Backblaze headline-vs-table definitional gap, not a pipeline discrepancy.
